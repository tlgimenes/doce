# Doce Constitution

## Core Principles

### I. Zero-Config First Run
The first launch MUST require no model picker, no API key entry, and no account
creation. The app MUST profile the host machine (RAM, chip generation, unified
memory, disk space) and automatically select and download an appropriate local
model, with progress shown and no configuration screen in the default path.
Advanced users MAY override the auto-selected model from settings, but this MUST
NOT be surfaced during onboarding. Any feature that adds a decision point to
first run (a picker, a prompt for credentials, a required account) is a
violation unless it ships behind an explicit "advanced" affordance outside the
default flow.

**Rationale**: Zero-config is Doce's entire reason to exist relative to
OpenClaw and other local-agent projects — it is the product's core
differentiator, not a nice-to-have.

### II. Local-By-Default Privacy
No telemetry and no accounts, by default. Nothing generated by or observed
about the user's usage leaves the device unless the user has explicitly opted
into a bridged channel (e.g. WhatsApp). Model inference, chat history,
workspace state, and permission grants MUST be stored and processed locally
(local SQLite, embedded inference). Any feature that would transmit user data
off-device MUST be opt-in, disclosed at the point of opt-in, and independent
of core chat/agent functionality.

**Rationale**: Local-by-default is the trust foundation the zero-config
onboarding depends on — users are asked to grant an agent file and shell
access before they've had time to build trust through configuration, so the
privacy guarantee has to be structural, not a settings toggle they must find.

### III. Native macOS Polish
Doce MUST feel like a first-class, signed and notarized Mac app, not a ported
CLI tool or Electron shell with rough edges. This applies to install/launch
experience, window chrome, permission prompts, and settings — all should match
platform conventions. Apple Silicon is the target architecture for v1.

**Rationale**: The product's positioning is explicitly against "CLI one-liner
plus onboarding flow" competitors; native polish is part of the zero-config
promise, not a separate cosmetic concern.

### IV. Explicit, Persistent Permissions
The agent operates freely only inside a workspace folder the user has
explicitly opened. The first time it wants to act outside that folder, or run
a new kind of shell command, it MUST trigger a plain-language approval prompt
with an "always allow this" option; trust decisions MUST persist per
workspace so approved action kinds do not re-prompt. Actions triggered by a
bridged channel (e.g. WhatsApp) are held to a stricter bar than actions
triggered from local chat: an inbound message MUST NEVER cause a shell command
or file write without a live, in-app approval on the Mac itself.

**Rationale**: Inbound messages from a bridged channel are the one place a
stranger could attempt prompt injection against an otherwise single-user,
fully local agent; the permission model must treat that as a distinct,
higher-risk trust boundary rather than reusing the local-chat bar.

### V. Extensibility via MCP and Skills
Doce MUST ship an MCP client so users can connect arbitrary MCP servers, and
MUST support filesystem-based `SKILL.md`-style capability packs (bundled
defaults + user-added) that the agent loop discovers and pulls into context
contextually. This mirrors the extensibility model of Claude Desktop/Code
rather than inventing a bespoke plugin system.

**Rationale**: MCP and skills are an established, growing ecosystem; adopting
existing conventions maximizes what Doce can use on day one and what users
already know how to extend.

### VI. v1 Scope Discipline
v1 targets macOS (Apple Silicon) only. The model marketplace/picker UI is NOT
the primary onboarding flow. Cloud sync, team/multi-user features, hosted
services, and RAG over arbitrary personal document stores are out of scope for
v1. Channel bridging beyond WhatsApp (iMessage, Telegram, Discord, Signal,
...) is explicitly deferred until WhatsApp validates demand for bridging at
all; WhatsApp bridging itself is v1.1 (fast-follow), not launch-blocking.
Expanding scope beyond these boundaries requires an explicit constitution
amendment, not an ad hoc feature decision.

**Rationale**: The product's differentiation depends on shipping the
zero-config core well rather than matching OpenClaw's channel breadth or
platform breadth at launch; scope creep here directly undermines the "sweet,
not complicated" positioning.

## Technology & Platform Constraints

- **Frontend**: React + TypeScript inside a Tauri webview.
- **Backend**: Rust. Inference via llama.cpp embedded through Rust bindings
  (not a spawned subprocess), a hardware profiler, a versioned/remotely
  refreshable model registry, a resumable checksum-verified downloader, an
  agent orchestrator (tool-use loop across built-in tools, MCP tools, and
  skills; GBNF-grammar-constrained tool calling for models without native
  function calling), a permission engine, an MCP client, a skills loader, and
  local SQLite storage for chat history, workspaces, settings, and permission
  grants.
- **Packaging & distribution**: Tauri bundler producing a signed, notarized
  `.dmg`; distributed via direct download and Homebrew cask. The Mac App
  Store is explicitly out of scope because shell execution and arbitrary file
  access are incompatible with App Sandbox.
- **WhatsApp bridging (v1.1)**: implemented via the `whatsapp-rust` crate,
  running in-process in the Rust backend (no bundled Node runtime). This is an
  unofficial protocol reimplementation; the associated ToS/ban risk MUST be
  disclosed clearly and un-skippably during WhatsApp setup, not buried in a
  ToS document.

## Development Workflow

- Feature work proceeds through the spec-kit workflow in order:
  constitution → specify → plan → tasks → implement, with `/speckit-clarify`,
  `/speckit-analyze`, and `/speckit-checklist` used as needed to de-risk
  ambiguity before planning and to validate consistency before implementation.
- Any feature touching onboarding, model selection, telemetry/data egress, or
  permission prompts MUST be checked against Principles I, II, and IV during
  `/speckit-plan`'s Constitution Check gate before implementation proceeds.
- Risk disclosures (e.g. WhatsApp ToS/ban risk) MUST be written into the spec
  and surfaced in-product before the corresponding feature ships, not added
  retroactively.

## Governance

This constitution supersedes ad hoc practice for any conflict between this
document and informal team decisions. Amendments require: a documented
rationale, an update to this file following the versioning policy below, and
propagation of any consequential changes to dependent templates
(`.specify/templates/*.md`) in the same change.

**Versioning policy**: MAJOR for backward-incompatible governance or
principle removals/redefinitions; MINOR for a new principle or materially
expanded guidance; PATCH for clarification, wording, or typo fixes.

**Compliance review**: Any `/speckit-plan` run MUST include a Constitution
Check against the principles above before design proceeds to task generation.
Deviations require explicit justification recorded in the plan's Complexity
Tracking section; unjustified deviations block progression to `/speckit-tasks`.
